## **1 Introduction**

This design document describes a technical solution for building an automated data ingestion pipeline on Alibaba Cloud based on GitHub Actions. By invoking Alibaba Cloud DataWorks OpenAPI, the solution automatically creates and configures data sources, MaxCompute tables, and data integration nodes, enabling automated ingestion of Parquet files from OSS into MaxCompute.

Core function：

· MaxCompute Table Creation

· Dataworks Data Integration Node Configuration

· Downstream Parquet File Deletion Node Configuration

· Parquet File Copy Node Configuration

· MaxCompute Partitioned Table Cleanup Node Configuration

### **2.2 Dataworks Workflow**

~~Upstream Node~~

Assignment Node

Node execution time: configured based on the Parquet file generation cycle, such as running once per hour or once per day.

Read all Parquet files in the directory and obtain the name of the latest uploaded file. Pass the value to the data integration node.

Default Configuration File: parquet-name-get.json

Script File: get-parquetname.py

**Data Integration Node**

Configure the node execution time based on the Parquet file generation cycle.

Import the data from the Parquet files in the camos/{feature_category} directory into the corresponding MaxCompute {feature_category} mapping table.

Configuration File: integration-config.json

**Python Node**

Confirm the status of the synchronization task. If it has been completed, move the Parquet files that have been synchronized to the "completed" directory.

Configuration File: parquet-file-completed.json

Script File: completed-parquet-file.py

**Python Node（MaxCompute Data deletion）**

Configure the execution time based on the data retention period.

Delete MaxCompute table partitions that are outside the retention period.

Configuration File: data-clean.json

Script File: clean-mc-table.py

### **2.3 MaxCompute Table Partition**

Partition naming: named according to the Parquet file generation cycle. For example, if a file is generated every hour, the partition name is in the format yyyymmddHH (year, month, day, and hour).

### **2.4 Node Creation**

A single DataWorks data integration node can synchronize only one type of feature data, that is, one type of Parquet file. Therefore, a separate data integration node must be configured for each feature data synchronization, and the feature Parquet files need to be stored in directories organized by feature type.

### **2.5 Parquet File Directory**

The Parquet files generated by DPC are uniformly stored in the feature directories of the OSS bucket under the CAMOS account, and they are moved into "completed" directory after the synchronization is completed. The completed Parquet files can be retained permanently or deleted based on a configured file expiration policy.

### **2.6 Node Update Strategy**

After the node is updated, there may be a maximum of 10 minutes of startup time. During this period, the tasks executed may not have any data. Please be reminded that don't worry, this data will be imported during the subsequent operation.

## **5 MaxCompute Multi-Environment DDL Execution**

MaxCompute does not have a built-in DDL version management mechanism by itself. Therefore, a set of DDL execution record and judgment mechanisms needs to be constructed at the engineering level to achieve unified deployment of DDL sources across multiple environments.

### **5.5 Data description after adding the column**

After adding a new field to the table, there will be no data of the newly added field in the historical data.

If historical data for new fields needs to be added, a new data integration node needs to be created and the historical Parquet files need to be regenerated for data synchronization.

## **6 MaxCompute Partition Table Cleanup**

· Each feature will automatically create the MaxCompute partition table cleaning node when creating the data integration node.

· The cleaning script for MaxCompute partitioned tables operates as follows:

o Under the feature directory, if there is a file named "{feature_category}-clean-mc-tables.py", then the cleaning script in the feature directory will be used. The operation personnel can customize and develop the cleaning mechanism as needed.

o In the feature directory, if there is no {feature_category}-clean-mc-tables.py file, then the default cleaning script in the scripts directory will be used.
